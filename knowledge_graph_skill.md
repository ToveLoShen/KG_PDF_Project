Skill name: pdf_to_course_knowledge_graph_xml

Purpose:
读取用户上传的 PDF 文档，按照课程知识图谱建模要求，从文档中抽取实体与关系，
生成一份结构清晰、节点不少于 40 个、输出格式为 XML 的知识图谱。
默认输出目标为可直接适配 KT-SQEP 类知识图谱工具的 XML，该知识图谱优先面向课程/教材/讲义/学习资料类 PDF，也可适配论文或综述类 PDF。

====================
一、总体建模原则
====================

1. 图谱中的实体分为两大类：
   - 独立实体
   - 附加实体

2. 独立实体又分为两类：
   - 内容型实体
   - 资源型实体

3. 本 skill 的核心任务是：
   - 从 PDF 中优先抽取内容型实体
   - 再从 PDF 中抽取 K、T、Z、Q、E、P 相关内容，按次序整理填入实体内容
   - 默认不输出资源型实体，除非用户明确要求保留
   - 默认直接生成工具兼容版 XML，不再先生成中间 graph-schema XML
   - 最终输出可直接导入工具的 XML

4. 输出图谱必须：
   - 节点总数不少于 40
   - 结构尽量层次清晰
   - 布局尽量美观
   - 同层节点尽量水平对齐
   - 同列节点尽量垂直对齐
   - 减少线条交叉
   - 保持阅读路径清晰

====================
二、实体体系
====================

A. 独立实体

A1. 内容型实体（必须优先抽取）
共有四种：

(1) 知识领域
- 代号：KA
- 图形：方形
- 背景色：红色
- 标记方式：实体左上角填入深红色字体“KA”
- 含义：课程的大主题、总领域、一级知识范围
- 抽取原则：通常来自 PDF 的总标题、章标题、一级主题

(2) 知识单元
- 代号：KU
- 图形：方形
- 背景色：绿色
- 标记方式：实体左上角填入深红色字体“KU”
- 含义：知识领域下的子模块、单元、章节部分
- 抽取原则：通常来自二级标题、章节内模块、方法分块

(3) 知识点
- 代号：KP
- 图形：方形
- 背景色：紫色
- 标记方式：实体左上角填入深红色字体“KP”
- 含义：某个具体概念、原理、方法点、步骤点
- 抽取原则：通常来自小节标题、定义、方法条目、概念说明

(4) 关键知识细节
- 代号：KD
- 图形：方形
- 背景色：蓝色
- 标记方式：实体左上角填入深红色字体“KD”
- 含义：对知识点的进一步展开，如公式、条件、性质、关键步骤、注意事项、对比点
- 抽取原则：通常来自正文中的定义细节、定理条件、关键结论、实现细节、例外情况

A2. 资源型实体
共有三种：

(1) 视频
- 代号：VD
- 图形：圆形
- 背景色：黄色
- 标记方式：实体中间填入深红色字体“VD”
- 含义：与内容相关的视频资源
- 抽取原则：仅当 PDF 中出现视频、讲解视频、演示视频、录像、视频课等信息时生成

(2) PPT
- 代号：PT
- 图形：圆形
- 背景色：黄色
- 标记方式：实体中间填入深红色字体“PT”
- 含义：与内容相关的课件资源
- 抽取原则：仅当 PDF 中出现课件、PPT、幻灯片、讲义展示页等信息时生成

(3) 文档
- 代号：PD
- 图形：圆形
- 背景色：黄色
- 标记方式：实体中间填入深红色字体“PD”
- 含义：与内容相关的文档资源
- 抽取原则：仅当 PDF 中出现文档、说明书、阅读材料、参考资料、论文、附录材料等信息时生成
- 默认目标工具 schema 中，资源型实体不作为默认输出内容
- 只有当用户明确要求保留资源节点时，才生成 VD / PT / PD

B. 附加实体
- 附加实体不单独脱离独立实体存在
- 作为对某个独立实体属性的补充说明
- 若 PDF 中没有明确可抽取的附加属性，则可省略附加实体
- 若需要，可用于记录：
  - 该节点对应的 K / T / Z / Q / E / P 标签
  - 页码来源
  - 重要程度
  - 是否为 inferred（推断得到）

====================
三、内容读取顺序
====================

必须严格遵守以下顺序：

第一步：先识别内容型实体
- 从 PDF 中先抽取 KA、KU、KP、KD
- 建立课程内容主干
- 保证图谱骨架先形成

第二步：默认跳过资源型实体
- 默认不生成 VD、PT、PD
- 若用户明确要求保留资源型实体，才从 PDF 中查找 VD、PT、PD
- 资源型实体只能作为补充，不能喧宾夺主

第三步：读取 K、T、Z、Q、E、P 相关内容
- 从 PDF 中提取如下六类内容：
  - K：知识性内容
  - T：思维、本质、原理、方法论类内容
  - Z：课程思政、价值导向、责任意识、伦理规范、工程规范类内容
  - Q：问题、提问、思考题、为什么、如何、辨析类内容
  - E：示例、案例、例题、应用场景
  - P：练习、习题、训练任务、作业要求

第四步：将 K、T、Z、Q、E、P 的内容按次序排列到对应实体内部
- 不再单独把 K、T、Z、Q、E、P 生成为外部独立资源节点
- 而是把这些内容作为内容型实体的内部组织内容
- 每个内容型实体内部必须按如下顺序填写内容：
  1. K：该实体对应的知识性内容
  2. T：该实体对应的思维/本质理解
  3. Z：若有，则加入思政/责任/价值内容
  4. Q：与该实体相关的问题
  5. E：与该实体相关的示例
  6. P：与该实体相关的练习

若某项缺失，可以省略该小项，不要强行编造。

====================
四、关系体系
====================

A. 内容型实体 与 资源型实体 的关系
- 连接方式：绿色虚线
- 箭头：在虚线靠近资源型实体时附上与虚线颜色一致的箭头
- 含义：内容实体关联某个资源实体
- 方向：内容型实体 -> 资源型实体

XML 中建议关系类型名：
- RESOURCE

B. 内容型实体之间的三种关系

(1) 包含关系
- 表现：黑色细线从头到尾连接
- 在线中间附上黑色箭头
- 含义：上位实体包含下位实体
- 典型例子：
  - KA 包含 KU
  - KU 包含 KP
  - KP 包含 KD
- XML 中建议关系类型名：
  - CONTAINS

(2) 次序关系
- 表现：蓝色细线从头到尾连接
- 在线末端附上蓝色箭头
- 含义：表示教学顺序、学习顺序、推导顺序、出现顺序
- 典型例子：
  - 先学某概念，再学某方法
  - 先学前置知识，再学后续知识
- XML 中建议关系类型名：
  - ORDER

(3) 关键次序关系
- 表现：黑色粗线从头到尾连接
- 在线末端附上黑色箭头
- 含义：表示非常关键、不可颠倒的核心顺序关系
- 典型例子：
  - 核心前置概念 -> 核心方法
  - 基础定义 -> 关键结论
  - 关键原理 -> 核心应用
- XML 中建议关系类型名：
  - KEY_ORDER

====================
五、实体抽取规则
====================

1. KA 抽取规则
- 从总标题、章标题、课程主题中抽取
- KA 数量不宜过多，通常 1~4 个
- 一个 PDF 若只有一个总主题，也至少生成 1 个 KA

2. KU 抽取规则
- 从 KA 下的章节、模块、方法部分中抽取
- KU 是主干结构的重要中层节点
- 通常应有 3 个以上 KU

3. KP 抽取规则
- 从 KU 下的概念、方法点、原理点中抽取
- KP 应成为图谱中的核心数量主体之一
- 通常应有 10 个以上 KP

4. KD 抽取规则
- 从 KP 下的细节说明、公式、性质、约束、步骤、条件、例外、注意事项中抽取
- KD 用于拉开层次深度，避免图谱只有标题骨架
- 通常应有 10 个以上 KD

5. VD / PT / PD 抽取规则
- 只在 PDF 有明确依据时生成
- 若无明确资源，也可少量依据上下文推断，但需标记 inferred
- 不可为凑数而大量虚构资源实体
- 默认输出时不生成资源型实体

====================
六、实体内容填写规则
====================

每个内容型实体应尽量包含以下结构化内容字段：

- content: 实体名称
- class_name: 知识领域:KA / 知识单元:KU / 知识点:KP / 关键知识细节:KD
- attach: 6 位编码，按 K / T / Z / Q / E / P 顺序标记对应内容是否存在
- K: ...
- T: ...
- Z: ...
- Q: ...
- E: ...
- P: ...

填写要求：

1. K
- 写该实体最核心的知识描述
- 应直接来自 PDF 原文或高度忠实归纳

2. T
- 写该实体对应的思维、本质、原理理解
- 包括“为什么这样做”“底层逻辑是什么”“与其他概念如何区分”

3. Z
- 仅在 PDF 中确有价值导向、思政、责任、伦理、规范内容时填写
- 没有则省略

4. Q
- 写与该实体直接相关的问题
- 可来自文中提问，也可基于文中知识结构归纳出合理教学问题
- 但不得脱离文本主题

5. E
- 写与该实体相关的例子、案例、例题或应用场景

6. P
- 写与该实体相关的练习、任务、思考训练点

顺序要求：
实体内部内容必须严格按照 K -> T -> Z -> Q -> E -> P 排列。

attach 编码要求：
- attach 必须为 6 位字符串
- 顺序固定为 K / T / Z / Q / E / P
- 有内容记为 1，无内容记为 0
- 例如：
  - `110111` 表示该实体含 K、T、Q、E、P，不含 Z
  - `111111` 表示六类内容都存在
  - `100000` 表示只有 K

====================
七、图谱规模与质量要求
====================

1. 节点数量要求
- 总节点数不少于 40
- 优先建议 45~70 个节点
- 其中内容型实体必须是主体

2. 推荐数量下限
- KA >= 1
- KU >= 3
- KP >= 10
- KD >= 10
- 默认不要求生成资源型实体

3. 边数量要求
- 总边数建议不少于 45
- 至少保证：
  - 大部分内容实体都有包含关系
  - 主干学习链路上存在次序关系
  - 核心链路上存在关键次序关系

4. 层次要求
- 至少形成三层以上结构深度
- 推荐形成：
  - KA -> KU -> KP -> KD

5. 连通性要求
- 尽量形成连通图
- 减少孤立节点
- 默认不生成资源节点

====================
八、布局与美观要求
====================

生成 XML 时应同时输出布局信息，以便图谱在目标工具中直接打开时不被遮挡。

布局原则：
1. 同一层级节点尽量水平对齐
2. 同一序列节点尽量垂直或横向均匀排列
3. KA 放在最上层或最左侧主入口位置
4. KU 在 KA 下方或右侧一层展开
5. KP 在 KU 下方或右侧继续展开
6. KD 靠近其所属 KP
7. 若用户明确要求保留资源型节点，资源节点放在其对应内容节点的外围，不要插入主干中间
8. 包含关系尽量保持树状展开
9. 次序关系尽量沿统一方向排列，例如从左到右或从上到下
10. 关键次序关系尽量突出，不被大量普通边遮挡
11. 节点间距尽量均匀
12. 减少交叉线
13. 避免内容文本过长导致视觉拥挤
14. 整张图不要贴边摆放，需预留顶部和左侧安全边距

面向默认工具的坐标建议：
- 最小 x 建议不小于 320
- 最小 y 建议不小于 180
- KA 通常放在第一行中央
- KU 放在第二行横向展开
- KP 放在其下方按列展开
- KD 放在更下方并尽量与所属 KP 对齐
- 若图整体偏左或偏上，应整体平移，而不是只改个别节点

默认工具 schema 至少应输出：
- x
- y

若同时生成中间图谱草稿，可额外保留：
- width
- height
- level
- column
- row

默认输出策略：
- 默认直接输出工具兼容版 XML
- 不再先输出 `<graph><nodes><edges>` 结构作为主结果
- 若用户明确要求保留中间 graph-schema 版本，才可额外生成一份中间 XML 草稿

====================
九、XML 输出要求
====================

输出必须为合法 XML。

默认必须输出如下工具兼容结构：

```xml
<?xml version="1.0" encoding="utf-8"?>
<KG>
  教学知识图谱
  <entities>
    <entity>
      <id>1</id>
      <class_name>知识领域:KA</class_name>
      <classification>内容方法型节点</classification>
      <identity>知识</identity>
      <level>一级</level>
      <attach>110111</attach>
      <opentool>无</opentool>
      <content>ERP与SCM引论</content>
      <K>...</K>
      <T>...</T>
      <Z></Z>
      <Q>...</Q>
      <E>...</E>
      <P>...</P>
      <x>920</x>
      <y>200</y>
    </entity>
    <entity>
      <id>2</id>
      <class_name>知识单元:KU</class_name>
      <classification>内容方法型节点</classification>
      <identity>知识</identity>
      <level>二级</level>
      <attach>110111</attach>
      <opentool>无</opentool>
      <content>什么是ERP和SCM</content>
      <K>...</K>
      <T>...</T>
      <Z></Z>
      <Q>...</Q>
      <E>...</E>
      <P>...</P>
      <x>400</x>
      <y>380</y>
    </entity>
  </entities>
  <relations>
    <relation>
      <name>包含</name>
      <headnodeid>1</headnodeid>
      <tailnodeid>2</tailnodeid>
      <class_name>包含关系</class_name>
      <mask>知识连线</mask>
      <classification>包含关系</classification>
      <head_need>内容方法型节点</head_need>
      <tail_need>内容方法型节点</tail_need>
    </relation>
    <relation>
      <name>包含</name>
      <headnodeid>2</headnodeid>
      <tailnodeid>3</tailnodeid>
      <class_name>次序：次序关系</class_name>
      <mask>知识连线</mask>
      <classification>次序关系</classification>
      <head_need>内容方法型节点</head_need>
      <tail_need>内容方法型节点</tail_need>
    </relation>
  </relations>
</KG>
```

字段约束：
- `class_name` 必须使用以下之一：
  - `知识领域:KA`
  - `知识单元:KU`
  - `知识点:KP`
  - `关键知识细节:KD`
- `classification` 固定为 `内容方法型节点`
- `identity` 固定为 `知识`
- `level` 映射固定为：
  - KA -> `一级`
  - KU -> `二级`
  - KP -> `归纳级`
  - KD -> `内容级`
- `opentool` 固定为 `无`
- `content` 仅放实体标题，不要把 K/T/Z/Q/E/P 拼进同一个文本块
- `K/T/Z/Q/E/P` 必须作为实体内部的独立子元素，并严格保持这个顺序
- `x/y` 必须为可用于布局的数值

关系映射约束：
- `CONTAINS` -> `class_name=包含关系`，`classification=包含关系`
- `ORDER` -> `class_name=次序：次序关系`，`classification=次序关系`
- `KEY_ORDER` -> 默认也映射为 `class_name=次序：次序关系`，`classification=次序关系`
- 默认不输出 `RESOURCE` 关系
- `name` 固定写 `包含`
- `mask` 固定写 `知识连线`
- `head_need` 和 `tail_need` 固定写 `内容方法型节点`

默认禁止事项：
- 不要把 `<graph><nodes><edges>` 作为默认最终输出
- 不要把 `content_ordered` 作为默认最终输出结构
- 不要先交付中间 schema 再等待二次转换，除非用户明确要求

====================
十、生成策略
====================

1. 先通读 PDF，识别：
- 标题
- 章节
- 小节
- 定义
- 原理
- 方法步骤
- 公式
- 图表说明
- 例题
- 练习
- 讨论
- 参考材料

2. 先建立内容骨架：
- KA -> KU -> KP -> KD

3. 再将 K/T/Z/Q/E/P 依次填入对应实体内部，并计算 attach

4. 按默认工具 schema 直接组装最终 XML
- 直接生成 `<KG><entities><relations>` 结构
- 直接写入工具兼容字段，不再额外走中间 graph-schema 转换

5. 默认不要补充资源实体
- 只有用户明确要求时，才生成 VD / PT / PD

6. 再建立关系：
- 先建 CONTAINS
- 再建 ORDER
- 再建 KEY_ORDER
- 默认不建 RESOURCE

7. 最后检查：
- 节点数是否 >= 40
- 层次是否清楚
- 是否有明显孤立节点
- 是否有过长文本块
- 布局是否基本整齐
- 最小 x 是否 >= 320
- 最小 y 是否 >= 180
- 每个实体是否都按 K/T/Z/Q/E/P 顺序输出
- attach 是否与 K/T/Z/Q/E/P 的实际存在情况一致

====================
十一、推断与约束
====================

1. 可以归纳，不可捏造核心知识
2. 可以推断少量教学问题 Q、示例 E、练习 P
   但必须与 PDF 内容高度一致
3. Z 不强制生成
4. 默认不输出资源实体
5. 当 PDF 内容较少时，应优先细化 KD，而不是乱增资源节点
6. 若 PDF 是论文：
   - KA 可对应研究主题
   - KU 可对应方法模块
   - KP 可对应关键机制
   - KD 可对应公式、条件、实验细节、局限性细节
7. 若 PDF 是教材/讲义：
   - 优先按章节结构直接映射

====================
十二、最终输出格式
====================

默认直接输出 XML，不输出摘要统计，不输出额外说明。

若用户明确要求“先给统计再给 XML”，才允许先输出：

GRAPH_SUMMARY:
- title:
- total_nodes:
- total_edges:
